# 🔀 Premerger · 前处理合并器

一个用户消息智能合并与中断重试插件。
它在用户消息到达 LLM 之前介入，将短时间内连续发送的多条消息合并为一条，并在用户发送新消息时自动中断正在进行的 LLM 请求、重新处理合并后的消息。

插件的核心目标是：让 LLM 始终基于用户完整的输入来生成回复，而不是对每条碎片消息单独响应。

---

## ✨ 功能概览

### ⏳ 防抖收集

收到用户消息后，插件会等待一段可配置的防抖时间再发起 LLM 请求。在此期间，用户发送的后续消息会被追加到缓冲区，防抖计时器重置。

这适用于用户习惯将一段话拆成多条消息发送的场景。

### ⚡ 中断重试

当 LLM 正在处理请求时，如果用户又发送了新消息，插件会立即中断当前 LLM 请求，将新消息合并到缓冲区，重新发起请求。

被中断的请求**全程无痕**：用户不会收到任何提示，对话历史不会记录，其他插件也不会感知到这次请求曾经存在。

### 🖼️ 多媒体支持

自动识别用户消息中的图片，与文本一起传递给 LLM。

### 🔀 消息合并

多条消息按可配置的分隔符合并为一条。默认使用换行符。

### 📋 指令放行

以配置的前缀（如 `/`）开头的消息会被识别为指令，不参与合并，直接放行。收到指令时会立即结束当前防抖会话并刷新缓冲区。

### 👥 按用户隔离

每个用户拥有独立的消息缓冲区和 LLM 请求，互不干扰。支持私聊和群聊（群聊默认关闭，可配置开启）。

## 🔄 工作流程

```
用户消息1 到达
    │
    ▼
┌─────────────────────────────────┐
│  防抖等待（可配置，默认 0.5 秒） │
│  期间新消息重置计时器            │
└─────────────────────────────────┘
    │ 计时结束
    ▼
┌─────────────────────────────────┐
│  合并缓冲区消息 → 发起 LLM 请求  │
└─────────────────────────────────┘
    │ LLM 正在处理
    │
    │ 用户消息2 到达
    ▼
┌─────────────────────────────────┐
│  中断当前 LLM 请求（无痕）       │
│  合并消息1 + 消息2               │
│  重新防抖 → 重新发起 LLM 请求    │
└─────────────────────────────────┘
    │ LLM 完成
    ▼
  发送回复，保存对话历史
```

## 🛠️ 主要配置项

| 配置项 | 说明 | 默认值 |
|--------|------|--------|
| `enable` | 是否启用前处理合并器 | `true` |
| `debounce_time` | 防抖等待时间（秒），设为 0 则不做防抖 | `0.5` |
| `merge_separator` | 消息合并分隔符 | `\n`（换行符） |
| `enable_private_chat` | 是否在私聊中启用 | `true` |
| `enable_group_chat` | 是否在群聊中启用 | `false` |
| `max_retry_count` | 单次对话最大中断重试次数 | `5` |
| `command_prefixes` | 指令前缀列表，这些消息不参与合并 | `["/"]` |

## ⚠️ 注意事项

1. **防抖阶段兼容性**：防抖阶段的消息合并通过重构事件实现，框架的完整管道（dayflow、记忆召回、postsplitter 等）正常执行。
2. **中断重试的局限**：中断重试时，被中断的响应会被丢弃（`stop_event()`），然后通过 `context.llm_generate()` 异步重新请求。重试的回复不经过框架的完整管道，因此 `on_decorating_result` 钩子的插件（如后处理分段器）不会处理重试的回复。
3. **API 费用**：中断 LLM 请求时，API 调用已经发出，费用已经产生。中断只是丢弃响应结果，不会撤销已发生的 API 调用。
4. **最大中断次数**：为避免无限中断循环，插件设置了最大中断重试次数（默认 5 次）。达到上限后，当前响应正常发送。
5. **工具调用**：中断重试时不支持 LLM 工具调用（Function Calling）。如需工具调用，请关闭本插件。
6. **防抖时间建议**：`0.5` 秒适合大多数场景。如果用户打字较慢，可以适当增加到 `1.0` ~ `2.0` 秒；如果希望尽快响应，可以设为 `0`。
